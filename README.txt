sco-ldcp v4
==========

Power-2 limb-darkening coefficients (g, h) for exoplanet transit
photometry, computed by trilinear interpolation of published
Claret & Southworth tables.

Owner: Ed Mullen, Sycamore Canyon Observatory


WHAT IT DOES
------------

Given T_eff, log g, and [M/H], scoldcp returns power-2 limb-darkening
coefficients for 24 filters (Johnson-Cousins UBVRI JHK, Sloan ugriz,
Stromgren uvby, Gaia G_BP/G/G_RP, Kepler, TESS, CHEOPS, CBB). Interpolation
is trilinear in the three stellar parameters; no extrapolation.

The power-2 law is I(mu)/I(1) = 1 - g(1 - mu^h), where mu = cos(theta).
The coefficient g is a limb-intensity amplitude and h is a curvature
exponent; they are the two parameters a transit tool's power-2 mode takes,
in the same slot the quadratic law's u1, u2 occupy -- but g and h do not
correspond to u1 and u2 individually.

Data sources:

    Tag         Reference                                       Filters                 Models
    ---------   ---------------------------------------------   ---------------------   ---------------
    CS22        Claret & Southworth 2022, A&A 664, A128         Gaia, Kepler, TESS,     ATLAS
                                                                SDSS, Stromgren,
                                                                Johnson-Cousins, 2MASS
    CS23        Claret & Southworth 2023, A&A 674, A63          same as CS22 plus       PHOENIX-COND
                                                                CHEOPS                  (M2 truncation)
    CBB power-2 Claret, Mullen & Gary CBB power-2 table         CBB                     ATLAS

Least-squares fitting method throughout. Microturbulent velocity xi = 2.0 km/s.
CS23 PHOENIX-COND tables use the M2 (midpoint) truncation method, which fits
the spherical-model profiles better than M1 across all three passband groups.

STARTUP CACHE
-------------

Parsing the raw tables takes a noticeable fraction of a second, which shows up
as cold-start latency after idle. To avoid this, the app loads a pre-parsed
pickle (data/tables.pkl, ~9 MB) at startup instead. A cache hit takes ~150 ms.

The cache is regenerated automatically if it's missing, corrupt, or older
than any of the source files. Run build_cache.py to regenerate it manually
after editing any source table; it verifies the per-file row counts (7695 for
each CS22 file and CBB, 823 for each CS23 file, at xi = 2.0 km/s) and confirms
the cache round-trips.


NOTES
-----

- Filter codes 'u' (Stromgren) and 'U' (Johnson) are case-sensitive.
- SDSS filter codes retain their '*' suffix: u*, g*, r*, i*, z*.
- Source tables are read by whitespace tokenisation, not fixed-column slicing.
  The CS23 12-band table (table10) has internally inconsistent ReadMe byte
  ranges; whitespace splitting is collision-free and is the correct approach.
- The CS23 Stromgren h-block is in standard u,v,b,y order; the VizieR ReadMe's
  apparent v/b label swap is a documentation typo, not a data swap (confirmed
  by external cross-check against CB2011 and CS22).
- CHEOPS is available only with the PHOENIX-COND model (CS23, solar metallicity
  only); there is no ATLAS CHEOPS table. The frontend offers only PHOENIX for it.
- CBB is available only with the ATLAS model; there is no PHOENIX power-2 CBB
  table. The frontend offers only ATLAS for it.
- All CS23 PHOENIX-COND tables are solar metallicity only; the [M/H] input is
  treated as 0.0 for those filter/model combinations.
- Stellar parameters can be supplied directly, or resolved from a planet name
  or TOI identifier via /api/resolve (NEA and ExoFOP, with committed fallback
  snapshots in data/).
